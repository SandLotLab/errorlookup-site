Frequently Asked Questions about HTTP Status Codes

This page addresses common questions developers have about HTTP response codes. It is organized by topic to help you quickly find the answers you need when debugging APIs, websites, microservices and serverless functions.

Basics

HTTP status codes are grouped by class. Each class communicates a high-level outcome for a request. Within classes, individual codes provide additional context.

  • 1xx Informational – The request was received and the process is continuing.
  • 2xx Success – The request was successfully received, understood, and accepted.
  • 3xx Redirection – Further action needs to be taken by the user agent to fulfill the request.
  • 4xx Client Error – The request contains bad syntax or cannot be fulfilled.
  • 5xx Server Error – The server failed to fulfill an apparently valid request.

To learn about all status codes in each class, visit the Status Codes Index.

Success Codes

Success codes indicate that the request succeeded. The most common success code is 200 OK. Some other helpful success codes include:

  • 201 Created – Indicates a new resource has been created. The Location header contains the URI of the new resource.
  • 204 No Content – Indicates the request succeeded but there is no message body in the response.
  • 206 Partial Content – Indicates that the server is delivering only part of the resource due to a range header sent by the client.

Success responses may contain data in the body, or they may indicate a resource has been created or modified.

Redirection Codes

Redirection codes instruct the client to make a new request to a different URI. Choosing the correct redirect response is important for user experience and SEO.

  • 301 Moved Permanently – Use this for a permanent URL change. Search engines will transfer ranking signals to the new URL.
  • 302 Found – A temporary redirect. Clients should continue to request the original URI in future.
  • 307 Temporary Redirect – Keeps the request method and body intact on redirect. Use for temporary moves in APIs that require method preservation.
  • 308 Permanent Redirect – Permanent redirect that also preserves the method and body. Useful when migrating API endpoints.

Improper use of temporary versus permanent redirects can confuse search engines and dilute SEO value. Learn more in our 301 vs 302 comparison guide.

Client Error Codes

Client error codes indicate that something is wrong with the request. They signal that the client needs to change or fix the request before it will succeed.

  • 400 Bad Request – The server cannot process the request due to invalid syntax or missing parameters.
  • 401 Unauthorized – The request lacks valid authentication credentials. Clients should authenticate and retry.
  • 403 Forbidden – Authentication succeeded but the authenticated client does not have permission to access the resource.
  • 404 Not Found – The requested resource cannot be found. Double-check the URL or confirm if the resource has moved.
  • 429 Too Many Requests – The client has sent too many requests in a given amount of time and should slow down or back off.

When a client error occurs, inspect the request method, URL, headers, and body for mistakes. Verify authentication and authorization, and check API documentation for parameter requirements.

Server Error Codes

Server error codes indicate that the server encountered an error processing a valid request. These errors are usually due to application logic, server misconfiguration, or issues with upstream dependencies.

  • 500 Internal Server Error – A generic error when the server fails to process a request unexpectedly. Check logs and upstream dependencies.
  • 502 Bad Gateway – A gateway or proxy received an invalid response from an upstream server. Investigate load balancers and backend services.
  • 503 Service Unavailable – The server is temporarily unable to handle the request due to overload or maintenance. Proper responses should include a Retry-After header.
  • 504 Gateway Timeout – A gateway or proxy did not receive a timely response from an upstream server. Check network connectivity and service latency.

Server errors should be logged and monitored. Consider implementing error tracking, circuit breakers, and graceful degradation to improve resilience.

SEO and Error Pages

Serving the correct status code is essential for search engines to index content properly. Here are a few best practices:

  • Return a 404 or 410 response for missing content rather than a 200 page with a custom message.
  • Use 301 redirects when moving content permanently. This passes ranking signals to the new URL.
  • Use 302 or 307 redirects for temporary changes, such as A/B testing or short-lived campaigns.
  • Monitor logs for recurring 4xx or 5xx errors, and fix broken links or misconfigured routes promptly.
  • In API design, choose status codes that accurately reflect the outcome of each request. Incorrect status codes can confuse clients and lead to poor developer experience.
